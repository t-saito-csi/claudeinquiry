## 1. システム概要

### 1.1 目的

大規模病院における問診業務をデジタル化し、患者・医療スタッフ双方の負担を軽減する。患者は来院前または来院時に問診票をデジタルで記入し、医師は診察前にAIが整理した症状サマリーを確認することで、診察の質と効率を向上させる。

### 1.2 対象施設・規模

| 項目 | 値 |
|------|----|
| 対象施設 | 中規模以上の総合病院 |
| 1日あたり来院患者数 | 200〜500人 |
| 最大同時接続数（患者） | 100セッション |
| 対応診療科（初期リリース） | 内科・外科/整形外科・産婦人科・小児科 |
| 対応言語 | 日本語のみ |

### 1.3 システム構成概要

```
【事前問診（来院前）】
患者スマートフォン/PC
    │ QRコード（メール・紙）
    ▼
患者向け問診SPA（React）
    │ HTTPS / JWT
    ▼
FastAPI バックエンド
    ├─ 問診フォームエンジン（動的分岐）
    ├─ Claude API（AI症状サマリー）
    ├─ NEC MegaOakHR連携（HL7 FHIR R4）
    └─ PDF生成
    │
    ▼
PostgreSQL（暗号化済み）/ Redis（セッション）

【来院時問診】
院内タブレット端末
    │ QRスキャン
    ▼
同上フロー

【スタッフ側】
医師用ダッシュボード（React）
受付スタッフ画面（React）
管理者画面（React）
    │ HTTPS / JWT（RBAC）
    ▼
FastAPI バックエンド
```

## 2. 用語定義

| 用語 | 定義 |
|------|------|
| 問診 | 患者が症状・病歴等を記入する医療情報収集プロセス |
| 問診セッション | QRコード発行から問診完了までの一連のトランザクション |
| QRトークン | 問診セッションと患者を紐付ける一時トークン（有効期限付き） |
| フォーム定義 | 診療科ごとの質問項目・分岐ロジックを定義したJSONスキーマ |
| 動的分岐 | 患者の回答に応じて次の質問セットが変化するロジック |
| AIサマリー | Claude APIが問診回答を解析して生成する症状要約・フラグ情報 |
| EHR | Electronic Health Record（電子カルテ）。本システムではNEC MegaOakHR |
| FHIR | Fast Healthcare Interoperability Resources。医療データ交換標準規格（R4版） |
| RBAC | Role-Based Access Control（ロールベースアクセス制御） |

## 3. ステークホルダーとロール

### 3.1 システム利用者

| ロール | 説明 | 主な操作 |
|--------|------|----------|
| `patient`（患者） | 問診票を記入するエンドユーザー | 問診フォームへの回答 |

## 4. 業務フロー

### 4.1 事前問診フロー（来院前）

```
[病院受付/予約システム]
    │
    │ 予約確定時にQRコード発行APIを呼び出す
    ▼
[本システム: QRトークン生成・保存]
    │ メール or 紙の予約票にQRコードを印刷して患者へ
    ▼
[患者]
    │ QRコードをスキャン（スマートフォン/PC）
    ▼
[患者向け問診SPA]
    ├─ QRトークンの有効性検証
    ├─ 診療科フォームの表示
    ├─ 共通項目（主訴・既往歴・アレルギー・服薬・生活習慣・保険同意）の記入
    ├─ 診療科固有項目の記入（動的分岐あり）
    └─ 確認・送信
    ▼
[FastAPI バックエンド]
    ├─ 回答の暗号化保存（PostgreSQL）
    ├─ Claude APIによるAIサマリー生成（非同期）
    └─ NEC MegaOakHRへFHIRリソース送信（非同期）
    ▼
[医師用ダッシュボード]
    └─ 患者一覧に問診完了として表示
```

### 4.2 来院時問診フロー（院内タブレット）

```
[患者が受付に到着]
    │
    ├─ パターンA: 事前問診済み → 受付スタッフが完了確認
    │
    └─ パターンB: 未記入 → 受付スタッフがQRコードを発行・印刷
                    │
                    ▼
            [院内タブレット端末]
                    │ QRスキャン
                    ▼
            [患者が院内で問診記入] ← LocalStorageで一時保存（ネットワーク断対策）
                    ▼
            [送信] → 以降は事前問診フローと同じ
```

### 4.3 医師の診察前確認フロー

```
[医師用ダッシュボードにログイン]
    │
    ▼
[本日の外来患者一覧（担当診療科でフィルタ）]
    │ 問診完了患者を選択
    ▼
[患者詳細画面]
    ├─ 問診回答全文（左カラム）
    ├─ AIサマリー（右カラム）
    │   ├─ 症状要約
    │   ├─ 要注意フラグ（アレルギー・服薬リスク等）
    │   ├─ 推奨確認事項
    │   └─ 前回来院時との差分分析
    └─ PDF出力ボタン
```

### 4.4 フォーム管理フロー（管理者）

```
[管理者ログイン]
    │
    ▼
[フォーム管理一覧]
    │ 診療科を選択・新規作成
    ▼
[フォームビルダー]
    ├─ 質問項目の追加・編集・削除
    ├─ 分岐ロジックの設定
    └─ プレビュー確認
    │
    ▼
[テスト公開 → 本番公開]
```

## 5. 機能要件

### 5.1 QRコード認証機能

#### 5.1.1 QRコード発行

| ID | 要件 |
|----|------|
| F-QR-001 | 予約システムまたは受付スタッフの操作により、患者に紐付いたQRコードを発行できること |
| F-QR-002 | QRコードには有効期限を設定すること（予約日当日 + 翌日0時まで） |
| F-QR-003 | 1患者につき複数の予約日に対して個別のQRコードを発行できること |
| F-QR-004 | QRコードは使用後（問診完了後）に自動失効すること |
| F-QR-005 | QRコードはPNG画像として出力できること（印刷・メール添付用） |

#### 5.1.2 QRコード認証

| ID | 要件 |
|----|------|
| F-QR-010 | QRコードをスキャンした際に、トークンの有効性（有効期限・使用済み）を検証すること |
| F-QR-011 | 無効なトークンの場合、患者に分かりやすいエラーメッセージを表示すること |
| F-QR-012 | 同一QRコードの多重使用を防止すること（Redisでの排他制御） |
| F-QR-013 | QRコードスキャン後はJWTセッショントークンを発行し、以降の通信に使用すること |

### 5.2 問診フォーム機能

#### 5.2.1 共通問診項目

全診療科共通で以下の項目を収集する。

**基本情報（EHRから自動取得または患者入力）**

| 項目 | 形式 | 必須 | 備考 |
|------|------|------|------|
| 氏名 | テキスト | ○ | EHRから自動入力（確認のみ） |
| 生年月日 | 日付 | ○ | EHRから自動入力（確認のみ） |
| 性別 | 選択（男性/女性/その他） | ○ | |
| 電話番号 | テキスト | ○ | |

**主訴・症状**

| 項目 | 形式 | 必須 | 備考 |
|------|------|------|------|
| 今日の受診理由（主訴） | テキスト（自由記述） | ○ | 最大500文字 |
| 症状の開始時期 | 選択（今日/数日前/1週間以上/1ヶ月以上） | ○ | |
| 症状の程度 | スケール（1〜10） | ○ | |

**既往歴・アレルギー**

| 項目 | 形式 | 必須 | 備考 |
|------|------|------|------|
| 既往歴 | チェックボックス複数選択 + 自由記述 | ○ | 高血圧・糖尿病・心疾患・脳疾患・がん・その他 |
| 手術歴 | 有無 + 詳細テキスト | ○ | |
| アレルギー（薬剤） | 有無 + 薬剤名テキスト | ○ | |
| アレルギー（食物・その他） | 有無 + 詳細テキスト | ○ | |
| 薬剤副作用歴 | 有無 + 詳細テキスト | ○ | |

**現在服用中の薬**

| 項目 | 形式 | 必須 | 備考 |
|------|------|------|------|
| 処方薬の服用 | 有無 | ○ | |
| 薬品名・用量 | テキスト（複数行） | 有の場合○ | お薬手帳の写真アップロードも可 |
| 市販薬・サプリメント | 有無 + 詳細テキスト | ○ | |

**生活習慣**

| 項目 | 形式 | 必須 | 備考 |
|------|------|------|------|
| 喫煙 | 選択（なし/あり/過去にあり） | ○ | あり→1日本数・期間を追加質問 |
| 飲酒 | 選択（なし/機会飲酒/毎日） | ○ | 毎日→量を追加質問 |
| 運動習慣 | 選択（ほぼ毎日/週2〜3回/ほとんどなし） | ○ | |

**保険・同意**

| 項目 | 形式 | 必須 | 備考 |
|------|------|------|------|
| 健康保険の種類 | 選択（国民健康保険/社会保険/後期高齢/その他） | ○ | |
| 個人情報取扱い同意 | チェックボックス | ○ | 未チェックは送信不可 |
| AI分析データ利用同意 | チェックボックス | ○ | AIサマリー生成への同意 |

#### 5.2.2 診療科別問診項目

**内科**

| 項目 | 形式 | 分岐条件 |
|------|------|----------|
| 発熱の有無 | 有無 | 有→最高体温・解熱剤使用を追加質問 |
| 咳・痰 | 有無 | 有→乾いた咳/湿った咳・血痰の有無を追加質問 |
| 腹痛の有無 | 有無 | 有→部位・性状（持続/波状）・吐き気・下痢・便秘を追加質問 |
| 胸痛・動悸 | 有無 | 有→安静時/労作時・放散痛の有無を追加質問 |
| めまい・頭痛 | 有無 | 有→性状・頻度・随伴症状を追加質問 |
| むくみ | 有無 | 有→部位・左右差を追加質問 |
| 体重変化 | 有無 | 有→増加/減少・期間・食欲の変化を追加質問 |

**外科・整形外科**

| 項目 | 形式 | 分岐条件 |
|------|------|----------|
| 受診の主な理由 | 選択（外傷/痛み/しこり/その他） | 外傷→発生状況・応急処置の有無を追加質問 |
| 痛みの部位 | 身体図（タップ選択） | - |
| 痛みの性状 | 選択（鋭い/鈍い/焼けるような/締め付け） | - |
| 痛みのタイミング | 選択（安静時/動作時/常時） | - |
| 痺れ・感覚障害 | 有無 | 有→部位・左右差を追加質問 |
| 発症からの経過 | 選択（急性/慢性） | - |
| 以前の受診・治療歴 | 有無 | 有→施設名・治療内容を追加質問 |

**産婦人科**

| 項目 | 形式 | 分岐条件 |
|------|------|----------|
| 最終月経日 | 日付 | - |
| 月経周期 | 選択（規則的/不規則）+ 日数 | 不規則→詳細を追加質問 |
| 妊娠の可能性 | 有無 | 有→妊娠週数・産科医受診の有無を追加質問 |
| 妊娠・出産歴 | 有無 | 有→妊娠回数・出産回数を追加質問 |
| 子宮・卵巣の既往 | 有無 | 有→疾患名・手術歴を追加質問 |
| 帯下（おりもの）の変化 | 有無 | 有→色・量・臭いを追加質問 |
| 性交痛 | 有無 | - |
| 乳房の症状 | 有無 | 有→しこり/痛み/乳頭分泌を追加質問 |

**小児科**

| 項目 | 形式 | 備考 |
|------|------|------|
| 患児の氏名 | テキスト | 保護者が記入 |
| 患児の生年月日 | 日付 | |
| 保護者との続柄 | 選択（父/母/祖父/祖母/その他） | |
| かかりつけ医 | テキスト | |
| 予防接種歴 | チェックボックス（BCG/麻疹/水痘等） | |
| 発達・発育の問題 | 有無 | 有→詳細テキストを追加質問 |
| きょうだいの症状 | 有無 | 有→症状を追加質問（感染症集団発生の把握） |
| 保育施設・学校での感染情報 | テキスト自由記述 | |

#### 5.2.3 動的分岐ロジック仕様

| ID | 要件 |
|----|------|
| F-FORM-001 | 回答値に応じて次の質問セットを動的に表示・非表示できること |
| F-FORM-002 | 分岐条件は「等しい/含む/以上/以下/空である」の比較演算子をサポートすること |
| F-FORM-003 | 複数条件の組み合わせ（AND/OR）をサポートすること |
| F-FORM-004 | 分岐によりスキップされた項目はAIサマリーに影響しないこと |
| F-FORM-005 | 患者が前の質問に戻って回答を変更した場合、以降の分岐を再評価すること |
| F-FORM-006 | 回答内容はLocalStorageに随時保存し、ネットワーク断後も復元できること |
| F-FORM-007 | 問診フォームはステップ形式で表示し、一度に表示する質問数を制限すること |

### 5.3 AIサマリー機能

| ID | 要件 |
|----|------|
| F-AI-001 | 問診回答の送信完了後、非同期でClaude APIを呼び出しサマリーを生成すること |
| F-AI-002 | サマリー生成が完了したら医師用ダッシュボードにリアルタイム通知すること（WebSocket or ポーリング） |
| F-AI-003 | AIサマリーは以下の4セクションで構成すること |
| F-AI-004 | 患者の氏名・生年月日等の個人識別情報はプロンプトに含めないこと |
| F-AI-005 | AIサマリーは「参考情報」である旨を画面上に常時明示すること |
| F-AI-006 | サマリー生成失敗時は医師に通知し、問診回答は直接確認できること |
| F-AI-007 | 同一患者の過去の問診データがある場合、前回との差分を分析すること |

**AIサマリーの出力形式**

```
【症状要約】
患者は〇〇日前から□□の症状を訴えている。症状の程度は10段階中△。

【要注意フラグ】
- ⚠️ アレルギー: ペニシリン系抗生物質（アナフィラキシー歴あり）
- ⚠️ 服薬: ワーファリン服用中（出血リスク）
- ⚠️ 緊急フラグ: 胸痛＋左腕への放散痛の訴えあり

【推奨確認事項】
1. 胸痛の性状・持続時間の詳細確認
2. 血圧・脈拍の即時測定
3. ECG検査の検討

【前回との差分（2025-12-15来院時比較）】
- 新規症状: 胸痛（前回なし）
- 改善: 腹痛（前回あり → 今回なし）
```

## 7. 外部インターフェース要件

### 7.1 NEC MegaOakHR連携（HL7 FHIR R4）

| 項目 | 内容 |
|------|------|
| 連携方式 | HL7 FHIR R4 REST API |
| 認証方式 | OAuth 2.0（クライアントクレデンシャル） |
| 送信タイミング | 問診完了後、非同期で送信（最大3回リトライ、指数バックオフ） |
| 使用リソース | `QuestionnaireResponse`（問診回答）、`Patient`（患者紐付け） |

**送信する FHIR リソース**

```json
// QuestionnaireResponse の例
{
  "resourceType": "QuestionnaireResponse",
  "status": "completed",
  "subject": { "reference": "Patient/{megaoak_patient_id}" },
  "authored": "2026-03-28T09:30:00+09:00",
  "questionnaire": "Questionnaire/naika-v1",
  "item": [
    {
      "linkId": "chief_complaint",
      "text": "今日の受診理由",
      "answer": [{ "valueString": "3日前から続く発熱と咳" }]
    }
  ]
}
```

**エラーハンドリング**

| ケース | 対応 |
|--------|------|
| MegaOakHR側タイムアウト | 指数バックオフでリトライ（最大3回） |
| 3回失敗 | 管理者にアラートメール送信、DBにエラー状態として記録 |
| 患者ID不一致 | 管理者に通知、手動対応を要求 |

### 7.2 Claude API連携

| 項目 | 内容 |
|------|------|
| 使用モデル | `claude-sonnet-4-6` |
| 呼び出し方式 | FastAPI BackgroundTasks（非同期） |
| タイムアウト | 60秒 |
| リトライ | 最大2回（指数バックオフ） |
| 入力トークン上限 | 8,000トークン/リクエスト |
| 個人情報除外 | 氏名・生年月日・電話番号はプロンプトから除外 |

**システムプロンプト（設計指針）**

```
あなたは医療情報整理の支援AIです。
以下のルールを厳守してください：
- 医療診断・治療の指示は絶対に行わないこと
- 患者の回答を客観的に整理・要約すること
- 要注意フラグは回答に明示されている情報のみ根拠とすること
- 推測・憶測に基づく判断を含めないこと
- 出力は必ず指定のJSON形式で返すこと
```
